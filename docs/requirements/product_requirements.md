# Продуктовые требования (Product Requirements Document)

## 1. Цель документа
Описать ключевые требования к образовательной платформе, которая служит объектом тестирования в рамках портфолио. Документ фокусируется на функциональности, доступности (WCAG 2.1 AA) и пользовательских сценариях.

## 2. Объект тестирования
**Название (условное):** EdTech Demo Platform  
**Тип:** Образовательная онлайн-платформа (LMS – Learning Management System)  
**Демо-сайт:** https://demoqa.com (учебный сайт, имитирующий каталог курсов и формы)

## 3. Основные функциональные блоки

### 3.1 Авторизация и регистрация
- Пользователь может зарегистрироваться, указав email и пароль (длина пароля: 8–20 символов, буквы и цифры).
- Пользователь может войти в систему с валидными учётными данными.
- При неверном пароле или email – сообщение об ошибке.
- Поля форм имеют явные текстовые метки (`<label>`).
- Кнопка «Войти» доступна с клавиатуры (Enter).

### 3.2 Каталог курсов
- Отображение списка курсов в виде карточек (название, описание, цена, рейтинг, кнопка «Подробнее»).
- Фильтрация по категориям (например, «Программирование», «Дизайн», «Маркетинг»).
- Поиск по названию курса.
- Пагинация (по 10 курсов на страницу).
- Фильтры и поиск полностью управляются с клавиатуры.
- При изменении списка курсов динамически обновляется ARIA live region для скринридера.

### 3.3 Страница курса/урока
- Отображение полной информации о курсе (описание, программа, преподаватель, цена).
- Кнопка «Записаться на курс» или «Купить».
- Если курс бесплатный – кнопка «Начать обучение».
- При переходе к урокам – навигация «Следующий урок/Предыдущий урок».
- Видео-контент имеет субтитры (опционально для демо).

### 3.4 Личный кабинет
- Просмотр профиля (имя, email, аватар).
- Редактирование профиля (имя, пароль, настройки уведомлений).
- Отображение списка купленных/записанных курсов с прогрессом в процентах.
- История платежей (для платных курсов).

### 3.5 Платежный flow (критичный сценарий)
- Выбор курса → переход на страницу оплаты → ввод платёжных данных (тестовый режим, без реальной оплаты) → подтверждение → получение доступа к курсу.
- Все поля платежной формы имеют чёткие подсказки.
- Кнопка «Оплатить» блокируется во время обработки запроса и имеет индикатор загрузки.

## 4. Требования к доступности (WCAG 2.1 Level AA)
- **Контрастность:** обычный текст – ≥4.5:1, крупный – ≥3:1.
- **Клавиатурная навигация:** все интерактивные элементы достижимы через Tab, фокус видимый, нет ловушек фокуса.
- **Скринридеры:** у всех изображений есть `alt`, формы имеют связанные `<label>`, структура заголовков логичная.
- **Адаптивность:** корректное отображение на mobile (320px), tablet (768px) и desktop (1280px+).

## 5. Нефункциональные требования (тестовый контекст)
- Платформа должна загружать основные страницы за ≤3 секунды при скорости интернета ≥5 Мбит/с.
- В консоли браузера (F12 → Console) не должно быть критических ошибок JavaScript.
- Система должна корректно обрабатывать невалидные входные данные (спецсимволы, очень длинные строки).
- При масштабировании страницы до 200% контент не обрезается и не накладывается.

## 6. Целевая аудитория (пользовательские роли)
- **Студент:** просматривает каталог, записывается на курсы, изучает уроки, отслеживает прогресс.
- **Преподаватель (если применимо):** создаёт курсы, загружает материалы, отвечает на вопросы.
- **Администратор (не тестируется):** управление пользователями, курсами, платежами.

## 7. Тестовые данные (примеры)
| Поле | Валидные данные | Невалидные данные |
|------|----------------|-------------------|
| Email | user@example.com | user@example (без домена), user@.com |
| Пароль | Pass1234 (8–20 символов) | Pass (менее 8), очень длинный (21+) |
| Имя | Иван, John | `<script>alert(1)</script>`, пустая строка |
| Цена (число) | 1000, 0.99 | -100, "abc" |
